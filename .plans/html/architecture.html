Foundation

Build a source-control-grade sync engine with an Electron control panel.

The daemon owns correctness. The desktop app owns product experience. The backend coordinates metadata. R2 stores encrypted immutable blob packs.

Boundary rule: Electron never directly mutates workspace state. It talks to the daemon, and the daemon is the only local writer for Devbox metadata.

System Shape

Electron UI
  -> local daemon API
Rust daemon
  -> watcher + scanner
  -> policy engine
  -> snapshot engine
  -> SQLite metadata
  -> local content-addressed cache
  -> sync client + encrypted object transport
Rust backend
  -> Postgres metadata
  -> R2 encrypted blob packs

Storage Responsibilities

SQLite

Local metadata

Projects, snapshots, manifests, chunks, operations, local identity, policies, conflicts, cursors, restore attempts.

Filesystem

Local blobs

Content-addressed chunk and pack files. SQLite points to them; it does not store code blobs.

R2

Remote blobs

Encrypted immutable blob objects behind a replaceable provider interface. Local filesystem provider is the current foundation.

MVP Interfaces

  • SnapshotStore: create, read, diff, restore snapshots.
  • BlobStore: put, get, head, delete encrypted blob packs.
  • PolicyEngine: classify include, ignore, block, encrypt, or rehydrate decisions.
  • GitAdapter: inspect repo state, verify materialization, and avoid dumb-syncing `.git`.
  • DaemonApi: status, projects, timeline, restore, settings, warnings, pause.

Auth Boundary

  • Local identity and key material support encrypted object sync foundations.
  • A local identity is not cloud authentication or account ownership proof.
  • The local/mock auth foundation models account session state, device approval, key envelopes, revocation, and device/project cursors in SQLite.
  • The local/mock materialization foundation proves encrypted snapshot publish/import into a second SQLite/cache context, deferred materialize cursor commit, and safe restore-engine apply.
  • Local high-confidence secret blocking prevents detected files from becoming uploadable snapshot blobs.
  • Local conflict records compare and persist divergent snapshot rows without source bytes.
  • Local sync preflight reconciles receiver cursors with local and incoming snapshots, refusing divergent local/mock import/materialization with conflict records and no cursor advance.
  • Production sign-in, hosted metadata, recovery, real object-storage credentials, explicit secret allow policy, automatic merge/apply resolution, and conflict UI remain deferred.
  • The alpha path starts with two devices, but the account/device model supports many approved devices over time.
Keep every layer replaceable except the trust model: immutable snapshots, explainable policy, safe restore.
