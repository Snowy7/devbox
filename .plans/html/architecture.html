Foundation

Build a source-control-grade sync engine with an Electron control panel.

The daemon owns correctness. The desktop app owns product experience. The backend coordinates metadata. R2 stores encrypted immutable blob packs.

Boundary rule: Electron never directly mutates workspace state. It talks to the daemon, and the daemon is the only local writer for bindhub metadata.

System Shape

Electron UI
  -> local daemon API
Rust daemon
  -> watcher + scanner
  -> live sync scan/publish/pull loop
  -> policy engine
  -> snapshot engine
  -> SQLite metadata
  -> local content-addressed cache
  -> sync client + encrypted object transport
Rust backend
  -> metadata API foundation for accounts/devices/projects/manifests/cursors/object grants
  -> SQLite for local/dev and Postgres for Railway hosted alpha
  -> shared-bucket R2 encrypted blob packs with account/project prefixes
Release evidence
  -> macOS/Linux alpha tool archives
  -> unsigned desktop bundle
  -> deterministic two-device smoke harness

Storage Responsibilities

SQLite

Local metadata

Projects, snapshots, manifests, chunks, operations, local identity, policies, conflicts, cursors, restore attempts.

Filesystem

Local blobs

Content-addressed chunk and pack files. SQLite points to them; it does not store code blobs.

R2

Remote blobs

Encrypted immutable blob objects behind a replaceable provider interface. Local filesystem and S3-compatible R2/S3/MinIO providers are the current foundations.

MVP Interfaces

  • SnapshotStore: create, read, diff, restore snapshots.
  • BlobStore: put, get, head, delete encrypted blob packs.
  • PolicyEngine: classify include, ignore, block, encrypt, or rehydrate decisions.
  • GitAdapter: inspect repo state, verify materialization, and avoid dumb-syncing `.git`.
  • DaemonApi: status, folders, timeline, restore, settings, warnings, pause.

Auth Boundary

  • Local identity and key material support encrypted object sync foundations.
  • S3-compatible encrypted blob transport can target R2, AWS S3, or MinIO without printing or storing raw credentials.
  • The hosted metadata API foundation models server-side accounts, devices, projects, published snapshot manifests, and compare-and-set cursors behind explicit mock-dev and account-session auth boundaries.
  • Publish/import/materialize can opt into in-process mock-dev hosted metadata for manifest lookup and server-side cursor compare-and-set.
  • The daemon live sync loop reuses snapshot, metadata, encrypted remote, and materialize foundations to publish current work and pull latest remote state with deterministic test controls.
  • A local identity is not cloud authentication or account ownership proof.
  • The local/mock auth foundation models account session state, device approval, key envelopes, revocation, and device/project cursors in SQLite.
  • The production-shaped auth boundary models external provider subject, verified email/domain, account id, session id, token hash, expiration, and revocation without OAuth.
  • Hosted metadata request handlers resolve bearer session tokens through the hosted session store and scope device/project/snapshot/cursor operations to the authenticated account.
  • The hosted metadata server can run as a single-instance alpha API with `/ready`, one-time invite login, bearer session status/logout, and mock-dev auth disabled by default.
  • Hosted metadata now models managed object credential leases and object-access grants with authenticated account/session lookup, project scope, R2/S3/MinIO-compatible provider references, canonical shared-bucket prefixes, capabilities, expiration, revocation, and rotation generation.
  • Hosted object transfer proxies encrypted object bytes through the metadata API for external testers, keeping raw bucket credentials on the server side.
  • The hosted metadata backend now supports Railway-shaped Postgres via DATABASE_URL while preserving SQLite for local/dev tests.
  • Production-shaped pairing recovery and rotation primitives model receiver-generated join/complete handoff, recovery grant references, device rotation intents, revocation, and key-envelope rotation generation without live recovery UX.
  • The local/mock materialization foundation proves encrypted snapshot publish/import into a paired second SQLite/cache context, decrypt without publisher DB access, deferred materialize cursor commit, and safe restore-engine apply.
  • Local high-confidence secret blocking prevents detected files from becoming uploadable snapshot blobs.
  • Local conflict records compare and persist divergent snapshot rows without source bytes.
  • Local sync preflight reconciles receiver cursors with local and incoming snapshots, refusing divergent local/mock import/materialization with conflict records and no cursor advance.
  • The private-alpha Electron shell reads redacted `BINDHUB_*` setup state for local paths, hosted session/project config, object-access lease, shared bucket prefix, pairing, live sync command state, and safety records.
  • Alpha release scripts package CLI/daemon/metadata tools and an unsigned desktop bundle; the deterministic smoke harness proves two-device pairing, fail-closed pending receivers, live publish, latest pull, and materialization.
  • OAuth/OIDC login, live recovery UX, signed installers, live Cloudflare/AWS object credential provisioning, automatic merge/apply resolution, paid/team/agent/Loom work, and multi-region/observability hardening remain deferred.
  • The alpha path starts with two devices, but the account/device model supports many approved devices over time.
Keep every layer replaceable except the trust model: immutable snapshots, explainable policy, safe restore.
